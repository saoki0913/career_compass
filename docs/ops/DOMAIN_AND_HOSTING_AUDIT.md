# ドメイン・ホスティング監査と解約チェック

**最終更新**: 2026-03-25

**この文書の目的**: `shupass.jp` まわりのドメイン、DNS、外部ホスティング契約を安全に棚卸しし、不要な契約を解約してよいか判断するための手順をまとめます。

**誰が読むか**: プロダクトオーナー、運用担当、インフラ設定を整理したい開発者。

---

## 先に結論

就活Pass の現行リポジトリは、少なくとも設計上は以下の構成を前提にしています。

- フロントエンド: Vercel
- AI バックエンド: Railway
- データベース: Supabase
- 認証: Better Auth + Google OAuth
- 本番 origin: `https://www.shupass.jp`

そのため、外部で発生している「お名前.com レンタルサーバー」の請求は、現行アプリ本体と無関係な旧契約、別用途、または未整理の残契約である可能性があります。

ただし、**請求が不要に見えても、メール、旧サイト、DB、cron、バックアップ用途が残っている場合は即解約しない**でください。

---

## 1. お名前.com で見る画面ごとの確認手順

### 1-1. ログイン

- `お名前.com Navi` にログインする
- URL: `https://navi.onamae.com/top`

### 1-2. 契約一覧

- `レンタルサーバー` の契約一覧を開く

確認する項目:

- 契約中サービス名
- 契約ID
- 対象ドメイン
- 契約開始日
- 次回更新日
- 自動更新の有無

### 1-3. 請求明細

- `ご請求明細` を開く

確認する項目:

- 何の料金か
- 月額か、まとめ払いか
- いつまで課金対象か
- 解約後に返金されない条件があるか

### 1-4. サーバー管理 / コントロールパネル

- `コントロールパネル` または `サーバー管理` を開く

確認する項目:

- Web 公開領域にファイルがあるか
- `public_html` 相当のディレクトリに何があるか
- 初期ファイルだけか、実運用ファイルがあるか
- WordPress、静的HTML、PHP アプリが置かれていないか

### 1-5. FTP アカウント

- `FTP` または `FTP アカウント` を開く

確認する項目:

- 利用中の FTP アカウントがあるか
- 複数人運用の痕跡がないか
- 現在もアップロードに使っていないか

### 1-6. データベース

- `データベース` を開く

確認する項目:

- MySQL / MariaDB が作成されているか
- DB 名が運用用途を示していないか
- WordPress などが接続していないか

### 1-7. メール

- `メール`、`メールアカウント`、`メール転送` などを確認する

確認する項目:

- 実運用のメールボックスがあるか
- `support@` `info@` `contact@` などを使っていないか
- 転送設定や自動返信が残っていないか

### 1-8. SSL

- `SSL` 設定画面を確認する

確認する項目:

- サーバー側で証明書を発行していないか
- 現在の公開サイトがその証明書に依存していないか

### 1-9. Cron / 定期実行 / バックアップ

- `Cron`、`定期実行`、`バックアップ` を確認する

確認する項目:

- バッチ処理が残っていないか
- バックアップ保管先として使っていないか
- 解約で消えると困るデータがないか

### 1-10. 利用状況 / アクセスログ

- `アクセスログ` や `利用状況` が見られる場合は確認する

確認する項目:

- 直近アクセスがあるか
- ボット以外の利用痕跡があるか
- まだ何かが稼働していないか

## 2. Cloudflare DNS の安全な確認手順

DNS は削除より先に、必ず棚卸しとバックアップを行います。

### 2-1. 対象ゾーンを開く

- Cloudflare にログインする
- 対象ゾーン `shupass.jp` を開く
- `DNS` 画面へ進む

### 2-2. まず全レコードを控える

削除や変更の前に、全レコードを別のメモへ保存します。

最低限控える項目:

- `Type`
- `Name`
- `Content`
- `Proxy status`
- `TTL`

### 2-3. レコード種別ごとに目的を切り分ける

優先して見る種別:

- `A` / `AAAA`: サイトや API の接続先
- `CNAME`: `www` やサブドメインの接続先
- `MX`: メール配送先
- `TXT`: SPF、DKIM、site verification
- `CAA`: 証明書制御
- `SRV`: 特殊用途

### 2-4. サイト本体に関係する代表パターンを確認する

一般に、Vercel での代表例は次のとおりです。

- `@` が `76.76.21.21`
- `www` が `cname.vercel-dns.com` 系

これ以外の古い IP や CNAME が混在している場合、旧ホスティングや別サービスの名残の可能性があります。

### 2-5. 削除前に用途を確認する

レコードを消す前に、そのレコードが何に使われているかを確認します。

- Web 用レコード: ブラウザまたは `curl -I` で応答確認
- メール用レコード: `MX` と `TXT` を重視
- 用途不明レコード: すぐ消さず保留

### 2-6. 変更は一度にやらない

安全な進め方:

1. 不要そうな `A` / `CNAME` を 1 件ずつ確認する
2. 削除または変更する
3. サイト表示、ログイン、OAuth、決済、メールを確認する
4. 問題がなければ次の候補へ進む

### 2-7. `MX` と `TXT` は慎重に扱う

サイトが Vercel で動いていても、メールだけ別サービス依存のことがあります。

特に次は安易に消さないこと:

- `MX`
- SPF (`TXT`)
- DKIM (`TXT` / CNAME)
- Google / Stripe / Search Console の verification 用 `TXT`

## 3. 解約してよい条件のチェックリスト

以下がすべて `Yes` なら、かなり安全に解約判断できます。

- お名前.com レンタルサーバーの公開領域に現行サイトのファイルがない
- WordPress や独自アプリが載っていない
- お名前.com サーバー上の DB を使っていない
- お名前.com サーバー上のメール機能を使っていない
- お名前.com サーバー上の cron / 定期実行 / バックアップを使っていない
- `shupass.jp` と `www.shupass.jp` が Vercel 側で正常応答する
- バックエンドが Railway 側で運用されている
- DB が Supabase 側で運用されている
- Cloudflare DNS に、お名前.com サーバー必須のレコードが残っていない
- Google OAuth や Stripe webhook の URL 先が、お名前.com サーバーではない
- 関係者へ「まだ使っている用途がない」確認が取れている
- 解約前に必要なファイルや設定をバックアップ済み

---

## 4. 解約しない方がいいサイン

以下のどれか 1 つでも当てはまる場合は、解約を保留します。

- お名前.com サーバーに実ファイルが残っている
- 実運用のメールアカウントが残っている
- DB が残っている
- 用途不明の cron や転送設定がある
- DNS に残っている古いレコードの用途を説明できない
- 他メンバー確認が取れていない

このリポジトリ上の前提構成は次のとおりです。

- フロントエンド: Vercel
- バックエンド: Railway
- データベース: Supabase
- 本番 URL: `https://www.shupass.jp`

参照:

このため、お名前.com レンタルサーバーの請求が来ていても、**現行アプリ本体がそのサーバー上で動いているとは限りません**。

---

## 6. 実務上の推奨フロー

1. お名前.com でサーバー内の実利用有無を確認する
2. Cloudflare DNS を全件棚卸しする
3. メール用途の有無を確認する
4. 関係者確認を取る
5. バックアップを取る
6. 不要レコードを段階的に整理する
7. 最後にサーバー契約を解約する

この順番を崩すと、サイト本体は無事でもメールや旧資産を落とす事故が起きやすくなります。
