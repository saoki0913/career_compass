```
AGENTS.md / CLAUDE.md  (Subagent Routing 正本)
        │
        ▼

        │
        ▼
.claude/skills/*/SKILL.md  (35 skill: description マッチで自律起動)

        │
        ▼
.claude/hooks/*.sh      (9 hook: Pre/PostToolUse, SessionStart, FileChanged, SubagentStart, Stop, SessionEnd)
.claude/settings.json   (hook 配線)
.mcp.json               (Claude Code 用 project scope MCP: playwright + notion)
.cursor/mcp.json        (Cursor 用 project scope MCP: playwright + notion)
.codex/config.toml      (Codex 用 setup / verification / MCP)
~/.claude/settings.json (user scope MCP: context7、リポジトリ外)
.claude/commands/*.md   (3 command: workflow 一括実行)
```

### 1.2 canonical と mirror の関係

- **canonical source**: `private/agent-pipeline/skills/*.md`
- **generator**: `scripts/agent-pipeline/sync-pipeline.mjs`
- **mirror 対象（上書きされる）**:
  - `.claude/skills/{name}/SKILL.md`
  - `.codex/skills/{name}/SKILL.md`
  - `.codex/commands/{name}.md`
  - `.cursor/rules/{name}.mdc`
  - `.agents/skills/{name}/SKILL.md`（canonical を含む一部のみ）
- すべての生成ファイルの冒頭には `<!-- Generated by scripts/agent-pipeline/sync-pipeline.mjs. Edit private/agent-pipeline/skills/*.md instead. -->` バナーが入る。

**ルール**: `.claude/skills/`, `.codex/`, `.cursor/` は原則として手で編集しない。canonical を編集してから `node scripts/agent-pipeline/sync-pipeline.mjs` を実行し、mirror 差分を確認する。ただし 1.2.1 節に明記した non-canonical 例外は除く。

### 1.2.1 canonical 管轄外の skill

次の 10 skill は `sync-pipeline` の生成対象ではなく、`.agents/skills/` を実体とする **non-canonical skill** として手動管理する。

- 既存 symlink: `prompt-engineer`, `rag-engineer`, `hybrid-search-implementation`, `ui-ux-pro-max`, `vercel-react-best-practices`
- 追加 symlink: `ai-product`, `better-auth-best-practices`, `security-review`, `frontend-design`, `codex-review`

管理ルール:

- `.claude/skills/<name>` は `../../.agents/skills/<name>` への symlink として維持する
- 実体の更新は `.agents/skills/<name>/SKILL.md` を編集して行う
- これらの skill には generated banner が付かない。`private/agent-pipeline/skills/` へ移すまで sync-pipeline は触らない
- 新しい non-canonical skill を追加する場合は、本節と 7.3 の PR チェックリストを同じ PR で更新する
- `.cursor/rules/career-compass-core.mdc` は generator 管轄外の手動 `alwaysApply` ルールとして管理する

### 1.3 user scope と project scope の境界

| スコープ | 設定ファイル | 対象 | リポジトリ管理 |
|---|---|---|---|
| project scope | `.mcp.json` | Claude Code 用 MCP（playwright, notion） | yes |
| project scope | `.cursor/mcp.json` | Cursor 用 MCP（playwright, notion） | yes |
| project scope | `.codex/config.toml` | Codex 用 setup / verification / MCP | yes |
| project scope | `.claude/settings.json` | hooks 配線 | yes |
| user scope | `~/.claude/settings.json` | context7 MCP | no（個人環境が正本） |

MCP 設定はツールごとの設定ファイルがそれぞれ正本であり、`.mcp.json` から自動生成はしない。本ドキュメントは差分管理のために「存在と用途」を一覧化する。user scope の設定はリポジトリには記録しない。

### 2.2 opus と sonnet の使い分け方針

- **opus**（11 agents）: 設計判断、品質審査、セキュリティ監査、プロンプト最適化、実装、検索品質など「間違うとコストが高い」領域。
  - architect, prompt-engineer, security-auditor, code-reviewer, fastapi-developer, nextjs-developer, ui-designer, database-engineer, rag-engineer, search-quality-engineer, product-strategist
- **sonnet**（2 agents）: テスト自動化、リリース運用など「反復速度が重要」な領域。
  - test-automator, release-engineer

### 2.3 委譲判断フロー

1. `AGENTS.md` の Subagent Routing 表で対象パスを検索し、マッチすればその agent に委譲する。
2. 複数領域をまたぐ（2 ファイル以上 + 責務境界を跨ぐ）場合は `architect` を起点にし、architecture-gate skill の結果に従って個別 agent に分解する。
3. PR 前の品質審査は `code-reviewer`、セキュリティ境界の変更は `security-auditor` を追加で回す。
4. docs-only / test-only / 局所的な文言修正 / 明らかな局所バグ修正は委譲を省略してよい（`AGENTS.md` の方針）。

### 2.4 サブエージェントの追加・変更手順

1. `.claude/agents/<name>.md` を作成。frontmatter に `name`, `description`, `model`, `tools` を書く（既存 13 ファイルの形式に合わせる）。
2. `AGENTS.md` と `CLAUDE.md` の Subagent Routing 表に行を追加する（両ファイルを byte-identical に保つ）。
3. 本ドキュメント 2.1 の表にも行を追加する。`関連 skill` 列は実在する skill のみ記載する。
4. session-orientation hook が `ls .claude/agents` から active subagents 一覧を生成しているため、Claude Code を再起動すれば新 agent が注入される。

### 4.3 command を新設すべきケース

次の条件をすべて満たす時だけ command を作る。通常は skill で十分。

- 逐次ステップのワークフローで順序を強制する必要がある
- 1 プロンプトで description マッチだけでは暴発を避けられない
- 実行可能な code block として Claude に手順を順番に踏ませたい（skill は description マッチで呼ばれるだけで、step 順序は自律判断）

| `post-edit-dispatcher.sh` | PostToolUse | `Edit\|Write` | exit 0（提案のみ） | prompt / 保守性 / schema / test の 4 系統リマインダを path ごとに集約 |
| `session-orientation.sh` | SessionStart | `startup\|resume` | exit 0（stdout 注入） | branch / git status / 直近 5 コミット / active subagents を session 開始時に context 注入 |
| `file-changed-lint.sh` | FileChanged | `package.json\|tsconfig.json\|...` | exit 0（提案のみ） | 高シグナルな JS/TS 系ファイル変更時に `npm run lint -- <path>` を自動実行 |
| `subagent-start-log.sh` | SubagentStart | matcher なし | exit 0（ログのみ） | subagent 起動を `docs/ops/agent-usage.log` に記録 |
| `stop-summary.sh` | Stop | matcher なし | exit 0（要約のみ） | 応答完了時に `git status --short` の要約を出す |
| `session-end-cleanup.sh` | SessionEnd | matcher なし | exit 0（cleanup のみ） | edit-count ファイルの削除と終了理由ログ |

### 5.3 各 hook の詳細

#### 5.3.1 `ui-preflight-reminder.sh`（PreToolUse, Edit|Write）

- **対象 path**: `src/components/**`, `src/app/**/(page|layout|loading).tsx`, `src/components/skeletons/**` など UI ファイル
- **動作**: exit 0（警告のみ）。標準エラー出力に UI Change Workflow のチェックリストを出す
- **推奨手順**: `npm run ui:preflight -- <route> --surface=marketing|product` → `npm run lint:ui:guardrails` → `npm run test:ui:review -- <route>`
- **相互参照**: [`docs/architecture/FRONTEND_UI_GUIDELINES.md`](../architecture/FRONTEND_UI_GUIDELINES.md), [`docs/testing/UI_PLAYWRIGHT_VERIFICATION.md`](../testing/UI_PLAYWRIGHT_VERIFICATION.md), `AGENTS.md` の UI Change Workflow 節

#### 5.3.2 `git-push-guard.sh`（PreToolUse, Bash）

- **対象コマンド**: `git push` を含む Bash コマンド全般
- **block（exit 2）**: `git push --force`, `git push -f`, `git push --force-with-lease` を含むコマンド
- **警告（exit 0）**: `git push ... main` または `git push ... develop`
- **スルー（exit 0）**: それ以外
- **release 正本**: `make deploy` / `make ops-release-check` → `make deploy-stage-all` / `scripts/release/release-career-compass.sh`
- **相互参照**: [`docs/ops/CLI_GUARDRAILS.md`](./CLI_GUARDRAILS.md), [`docs/release/PRODUCTION.md`](../release/PRODUCTION.md)

#### 5.3.3 `secrets-guard.sh`（PreToolUse, Read|Bash）

- **block（exit 2）**:
  - Read tool で `codex-company/.secrets/` を含む path を開こうとする
  - Bash tool で `cat|head|tail|less|more|bat|sed|awk|grep|rg` 等の読み取り系コマンドが `codex-company/.secrets/` を対象にする
- **許可される操作**: `zsh scripts/release/sync-career-compass-secrets.sh --check`
- **相互参照**: [`docs/ops/SECURITY.md`](./SECURITY.md), [`docs/release/ENV_REFERENCE.md`](../release/ENV_REFERENCE.md), `CLAUDE.md` の API / Error Handling Rules 節

#### 5.3.4 `post-edit-dispatcher.sh`（PostToolUse, Edit|Write）

- **入力**: `file_path` と `session_id` を受け取り、1 回の hook 実行で複数の path-aware reminder を判定する
- **内包するリマインダ**:
  - prompt / LLM ファイル変更時の eval + 参考 ES 漏洩チェック
  - `.ts`, `.tsx`, `.py` 編集 5 回ごとの maintainability チェック
  - `src/lib/db/schema.ts` 変更時の migration 生成 / SQL レビュー
  - Vitest / Playwright / pytest テストファイル変更時の実行コマンド提案
- **発火頻度**: maintainability 用カウンタは `$HOME/.claude/sessions/career_compass/edit-count-{SESSION_ID}` で管理する
- **動作**: exit 0（提案のみ）。複数条件に同時一致する場合は同一実行内で順番にメッセージを出す

#### 5.3.5 `session-orientation.sh`（SessionStart, startup|resume）

- **動作**: exit 0。stdout が session 開始時の context に注入される（公式仕様）
- **注入内容**: branch 名、`git status --short` の先頭 20 行、`git log --oneline -5`、`ls .claude/agents` から生成した active subagents 一覧
- **運用上の効果**: 新しい agent を `.claude/agents/` に追加すると、次の session から自動で routing 候補として提示される

#### 5.3.6 `file-changed-lint.sh`（FileChanged）

- **対象 watch list**: `package.json`, `tsconfig.json`, `eslint.config.mjs`, `playwright.config.ts`, `playwright.live.config.ts`, `next.config.ts`, `next.config.js`, `src/lib/db/schema.ts`
- **動作**: exit 0（通知のみ）。対象ファイルが JS/TS 系なら `npm run lint -- <relative-path>` を実行し、結果を stderr に出す
- **制約**: Claude Code の `FileChanged` は literal filename watch のため、repo 全体の全ファイル監視ではなく高シグナルなファイルに限定している

#### 5.3.7 `subagent-start-log.sh`（SubagentStart）

- **入力**: `agent_type`, `agent_id`, `session_id`
- **動作**: exit 0。`docs/ops/agent-usage.log` に UTC timestamp 付きで追記する
- **用途**: subagent 利用頻度とコスト傾向の事後レビュー

#### 5.3.8 `stop-summary.sh`（Stop）

- **動作**: exit 0。branch 名と `git status --short` の要約を stderr に出す
- **用途**: 応答完了ごとに dirty 状態を可視化し、未整理差分の持ち越しを減らす

#### 5.3.9 `session-end-cleanup.sh`（SessionEnd）

- **対象**: `$HOME/.claude/sessions/career_compass/edit-count-{SESSION_ID}`
- **動作**: exit 0。session 単位の maintainability counter を削除し、終了理由を stderr に出す
- **制約**: `SessionEnd` はデフォルト timeout が短いため、重い処理は入れない

### 5.4 dry-run 検証手順

```bash
# UI ファイル編集前リマインダが発火するか
echo '{"tool_input":{"file_path":"/tmp/src/components/Foo.tsx"}}' \
  | .claude/hooks/ui-preflight-reminder.sh; echo "exit=$?"
# 期待: stderr にチェックリスト、exit=0

# git push --force が block されるか
echo '{"tool_input":{"command":"git push --force origin main"}}' \
  | .claude/hooks/git-push-guard.sh; echo "exit=$?"
# 期待: stderr に block メッセージ、exit=2

# main への push が警告のみで通るか
echo '{"tool_input":{"command":"git push origin main"}}' \
  | .claude/hooks/git-push-guard.sh; echo "exit=$?"
# 期待: stderr に警告、exit=0

# secrets Read が block されるか
echo '{"tool_name":"Read","tool_input":{"file_path":"codex-company/.secrets/.env"}}' \
  | .claude/hooks/secrets-guard.sh; echo "exit=$?"
# 期待: stderr に block メッセージ、exit=2

# secrets の cat が block されるか
echo '{"tool_name":"Bash","tool_input":{"command":"cat codex-company/.secrets/.env"}}' \
  | .claude/hooks/secrets-guard.sh; echo "exit=$?"
# 期待: stderr に block メッセージ、exit=2

# schema.ts 変更リマインダが発火するか
echo '{"tool_input":{"file_path":"src/lib/db/schema.ts"}}' \
  | .claude/hooks/post-edit-dispatcher.sh; echo "exit=$?"
# 期待: stderr に migration リマインダ、exit=0

# pytest テスト変更リマインダが発火するか（多段パス）
echo '{"tool_input":{"file_path":"backend/tests/es_review/test_reference_es_quality.py"}}' \
  | .claude/hooks/post-edit-dispatcher.sh; echo "exit=$?"
# 期待: stderr にファイルパス埋め込みのテスト実行提案、exit=0

# FileChanged lint が走るか
echo '{"file_path":"'"$PWD"'/package.json"}' \
  | .claude/hooks/file-changed-lint.sh; echo "exit=$?"
# 期待: stderr に lint 実行結果、exit=0

# SessionEnd cleanup が走るか
echo '{"session_id":"dry-run","reason":"other"}' \
  | .claude/hooks/session-end-cleanup.sh; echo "exit=$?"
# 期待: stderr に cleanup 結果、exit=0
```

### 5.5 新しい hook を追加する手順

1. `.claude/hooks/<name>.sh` を作成し、`chmod +x` で実行権限を付与する
2. 入力は stdin から JSON で来る。`jq -r '.tool_input.*'` または `.tool_name` を使って判定する
3. 警告は stderr + exit 0、block は stderr + exit 2
4. `.claude/settings.json` の該当 event 配下に `matcher` と `command` を追加する
5. 5.1 の一覧表と 5.3 の詳細節に追記する
6. 5.4 の手順で dry-run する

## 6. MCP サーバー リファレンス

### 6.1 project scope MCP 設定

**現行**: Claude Code 用 `.mcp.json` と Cursor 用 `.cursor/mcp.json` の 2 系統。どちらも playwright（stdio）と notion（http）の 2 サーバーを持つ。

- **正本ポリシー**: `.mcp.json` と `.cursor/mcp.json` はそれぞれ独立した設定ファイルとして管理する。片方から他方を自動生成しない
- **変更時ルール**: MCP サーバーの追加・削除・接続先変更をした PR では、両ファイルの整合と本節の更新を同じ PR で行う

#### Claude Code: `.mcp.json`

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@playwright/mcp", "--headless"],
      "env": {}
    },
    "notion": {
      "type": "http",
      "url": "https://mcp.notion.com/mcp"
    }
  }
}
```

#### Cursor: `.cursor/mcp.json`

内容は `.mcp.json` と同一の 2 サーバーを持つ。Cursor 側でも Playwright / Notion を同じ project scope で利用できるようにするための設定。

#### playwright

- **用途**: `ui-designer` / `test-automator` からの interactive な browser 操作。`mcp__playwright__*` ツール群が提供される
- **承認**: 初回呼び出し時に MCP 承認プロンプトが出る。ユーザ承認後にツールが有効化される
- **相互参照**: [`docs/testing/UI_PLAYWRIGHT_VERIFICATION.md`](../testing/UI_PLAYWRIGHT_VERIFICATION.md)

#### notion

- **用途**: Notion Prompt Registry と参考 ES Database の取得。prompt 同期 (`npm run prompts:sync`) や参考 ES 取り込みで raw JSON 取得に使う
- **type**: http（Notion 公式 MCP エンドポイント）
- **認証**: 初回接続時に Notion OAuth フローが走る。個人の Notion アカウントで承認する
- **相互参照**: [`docs/setup/MCP_SETUP.md`](../setup/MCP_SETUP.md) の「Notion 参考 ES の取り込み」「Notion Prompt Registry の同期」節

### 6.2 user scope: `~/.claude/settings.json`（リポジトリ外）

**設定内容**: context7 MCP（`mcp__context7__*` ツール群）。

- **用途**: 全プロジェクト共通で Claude / OpenAI / FastAPI / Next.js / Drizzle などライブラリドキュメントの注入
- **管理**: 個人環境が正本。リポジトリでは設定 JSON を管理しない
- **導入・再設定**: 個別の環境で `~/.claude/settings.json` を編集する。リポジトリへ commit しない
- **外部参照**: https://code.claude.com/docs/en/mcp

### 6.3 導入しない MCP と理由

次の MCP サーバーはあえて導入していない。理由は CLI で代替可能だから。

| サーバー | 代替手段 |
|---|---|
| Supabase MCP | `supabase` CLI（`docs/ops/CLI_GUARDRAILS.md` の許可操作に沿う） |
| Railway MCP | `railway` CLI + `scripts/release/*` |
| Vercel MCP | `vercel` CLI + `scripts/release/*` |
| GitHub MCP | `gh` CLI（`gh pr create` / `gh pr merge` は release scripts 経由） |
| Stripe MCP | `stripe` CLI + webhook ローカル listen |

agent system prompt から CLI を直接叩く方が、権限モデルと監査ログが既存の CLI Guardrails で完結するため、現時点では MCP 化しない判断。将来追加するかは使用頻度と代替 CLI の限界で判断する。

### 6.4 MCP を追加・変更する手順

- **project scope**（リポジトリ共通）:
  1. `.mcp.json`, `.cursor/mcp.json`, `.codex/config.toml` の該当箇所を同じ PR で更新する
  2. 本ドキュメント 6.1 の節を更新する
  3. 初回起動時に承認プロンプトが出る。承認する
- **user scope**（個人環境）:
  1. `~/.claude/settings.json` の `mcpServers` に server を追記する
  2. リポジトリには commit しない
  3. 本ドキュメント 6.2 の節には「存在と用途」のみ反映する

### 6.5 意図的に未導入の改善

- **opus → sonnet 再評価**: 現時点では `opus` の割当を維持する。品質劣化計測の dogfood 設計が別途必要
- **HTTP hook**: 通知先（Slack / Discord）が未整備のため導入しない
- **Prompt hook**: 大規模ファイル編集前の LLM 判定は有用だが、常時コスト増を伴うため現時点では保留

## 7. 運用ガイド

### 7.1 agent 委譲の判断

| 状況 | 対応 |
|---|---|
| routing 表にマッチ、影響範囲 2 ファイル以上 | 該当 agent に委譲 |
| routing 表にマッチ、影響範囲 1 ファイル未満、リスク低 | 直接編集可 |
| 複数領域にまたがる、責務境界を跨ぐ | `architect` 起点、`architecture-gate` で判定してから分解 |
| PR 前の品質審査 | `code-reviewer` |
| auth / billing / webhook / csrf の変更 | `security-auditor` を必ず併用 |
| docs-only / test-only / 局所文言 / 局所バグ | 委譲省略可 |

### 7.2 skill の自律起動が動かないときの診断

1. description マッチを阻害する typo が `.claude/skills/<name>/SKILL.md` に入っていないか確認する
2. 他の skill との trigger 衝突がないか、description 文を比較する
3. `node scripts/agent-pipeline/sync-pipeline.mjs` を走らせて mirror が stale でないか確認する
4. canonical (`private/agent-pipeline/skills/*.md`) が正しいか確認する
5. Claude Code を再起動して session-orientation hook のコンテキストを更新する

### 7.3 ハーネス変更の PR チェックリスト

- [ ] canonical → generator → mirror の順で commit されている
- [ ] `.claude/skills/*.md`, `.codex/**/*.md`, generator 管轄の `.cursor/rules/*.mdc` を手編集していない（例外: 1.2.1 の non-canonical skill と `career-compass-core.mdc`）
- [ ] `AGENTS.md` と `CLAUDE.md` が byte-identical（`diff AGENTS.md CLAUDE.md` が空）
- [ ] `.claude/hooks/*.sh` を変更した場合、5.4 の dry-run が通る
- [ ] `.mcp.json`, `.cursor/mcp.json`, `.codex/config.toml` を変更した場合、6.1 が更新されている
- [ ] `docs/ops/AI_HARNESS.md` の関連節と「最終確認日」が更新されている
- [ ] `npm run test:agent-pipeline` が通る

### 7.5 トラブルシューティング

| 症状 | 対応 |
|---|---|
| skill が自律起動しない | 7.2 を順番に確認。generator 未実行が最も多い |
| hook が発火しない | `chmod +x .claude/hooks/*.sh`、`.claude/settings.json` の matcher を確認 |
| hook が誤検知する | 5.4 の dry-run で stdin JSON を再現して原因を特定 |
