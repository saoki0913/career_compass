# AIエージェント開発レビュー・チェックリスト（優先度順）

AIエージェント、AIコーディングエージェント、LLM組み込み機能、RAG、ツール実行型エージェントを含む開発全般で使うためのレビューリストです。

## 使い方

- **P0 / Blocker**：未対応ならマージ・本番反映しない。
- **P1 / High**：本番前に原則対応する。未対応なら明確な理由と代替策を書く。
- **P2 / Medium**：安定運用・品質のために対応する。
- **P3 / Low**：保守性・運用品質を上げるために対応する。
- **P4 / Ongoing**：リリース後も継続的に確認する。

AIエージェントによる実装でも、通常のソフトウェア開発と同じく、**認証・認可、データ保護、仕様適合、テスト、運用、保守性**を確認します。加えて、AIエージェント特有の **プロンプト、ツール権限、自律実行、ログ、評価、ガードレール**を必ず確認します。

---

# P0 / Blocker：未対応ならマージしない

## 1. AIエージェントの実行範囲・自律性

- [ ] AIエージェントが依頼範囲外のファイル・仕様・機能を変更していない。
- [ ] AIエージェントが未依頼の機能追加、仕様変更、削除を行っていない。
- [ ] 破壊的操作、削除、決済、メール送信、本番デプロイ、DB migration などは人間承認が必要になっている。
- [ ] AIエージェントに渡すツール権限が最小権限になっている。
- [ ] 開発用AIエージェントが本番DB、本番secret、本番APIキーに直接アクセスできない。
- [ ] AIエージェントがshell commandを実行する場合、危険コマンドや本番操作が制限されている。
- [ ] AIエージェントが生成したコード、SQL、shell、設定ファイルをそのまま本番実行していない。
- [ ] AIエージェントのツール実行結果、判断理由、変更対象が追跡可能になっている。
- [ ] AIエージェントの出力を信頼せず、重要処理ではサーバー側で検証している。
- [ ] AIエージェントが外部入力やWebページの指示をsystem/developer指示より優先しない設計になっている。

## 2. 認証・認可・データ分離

- [ ] 未ログインで保護ページ・保護APIにアクセスできない。
- [ ] ログイン済みでも権限外のデータ取得・作成・更新・削除ができない。
- [ ] サーバー側で認可チェックを行っており、フロント側の表示制御だけに依存していない。
- [ ] URL、ID、slug、uuid を書き換えて他人のデータにアクセスできない。
- [ ] 組織、チーム、テナント、ワークスペース、ユーザー単位の境界が守られている。
- [ ] 一般ユーザーが管理者APIや管理者画面を実行できない。
- [ ] 削除済み、停止済み、退会済みユーザーが引き続きアクセスできない。
- [ ] JWT、session、cookie、access token の期限・署名・発行元を検証している。
- [ ] refresh token や長期認証情報が安全に保存・ローテーションされる。
- [ ] DB側のRow Level Security、policy、constraintなどを使う場合、実装と一致している。

## 3. Secret・個人情報・ログ

- [ ] APIキー、DB URL、秘密鍵、OAuth secret、LLM API key がGitに含まれていない。
- [ ] secret がフロントエンドbundle、ブラウザconsole、HTML、public envに漏れていない。
- [ ] secret、token、cookie、authorization header がアプリログに出力されていない。
- [ ] 個人情報、機密情報、決済情報、認証情報が不要にログ保存されていない。
- [ ] AIへのprompt、response、tool resultのログに機密情報が残らないように制限・マスクされている。
- [ ] 外部LLM APIへ送る個人情報・機密情報が最小化されている。
- [ ] AIエージェントのメモリ、RAG index、vector DBに不要な個人情報を保存していない。
- [ ] ユーザーが削除したデータがAIメモリ、検索index、ログ、キャッシュに残り続けない。
- [ ] .env.example などのサンプルファイルに本物の値が混ざっていない。
- [ ] 本番secretをローカル開発やテスト実行に使っていない。

## 4. 重大な脆弱性

- [ ] SQL Injectionが起きないようにparameterized queryやORMの安全なAPIを使っている。
- [ ] XSSが起きないようにユーザー入力・AI生成テキストを安全にescape/sanitizeしている。
- [ ] CSRF対策が必要な認証方式ではCSRF tokenやSameSite設定がある。
- [ ] SSRFが起きないようにURL取得・webhook・crawler・agent browserの接続先を制限している。
- [ ] Path Traversalにより想定外のファイルを読めない。
- [ ] Command Injectionによりユーザー入力やAI出力がshell commandへ危険に渡らない。
- [ ] File uploadで拡張子、MIME type、サイズ、保存先、公開範囲を検証している。
- [ ] unsafe deserialization、eval、Function constructor、動的importの危険利用がない。
- [ ] 依存ライブラリに既知の重大脆弱性がない。
- [ ] 管理画面、internal API、debug endpointが外部公開されていない。

## 5. 環境・本番安全性

- [ ] production、staging、development のDB・API・storage・queueが分離されている。
- [ ] stagingやpreview環境が本番DBを書き換えない。
- [ ] 本番でtest key、sandbox key、mock endpointを使っていない。
- [ ] 開発環境で本番決済、本番メール送信、本番通知が実行されない。
- [ ] 環境変数が未設定の場合、安全に失敗する。危険なdefault値で起動しない。
- [ ] CORSが本番で `*` や不要なoriginを許可していない。
- [ ] CSP、セキュリティヘッダー、cookie属性が本番要件に合っている。
- [ ] 外部API、SDK、決済、LLM APIのversionが固定または明示されている。
- [ ] preview deployでsecretや内部情報が第三者に露出しない。
- [ ] CI/CDのsecretがログやartifactに出ていない。

## 6. データ破壊・二重実行・DB変更

- [ ] 作成、更新、削除、決済、通知、メール送信などの重要操作に冪等性がある。
- [ ] ボタン連打、再送、retry、webhook重複で二重保存・二重課金・二重通知が起きない。
- [ ] 複数DB更新はtransactionで保護され、途中失敗で不整合にならない。
- [ ] migrationがアプリコードと整合しており、適用順序が安全である。
- [ ] migration失敗時や本番障害時にrollbackまたはforward fixの手順がある。
- [ ] cascade deleteで消してはいけない関連データを削除しない。
- [ ] 論理削除と物理削除の扱いが一貫している。
- [ ] UNIQUE、NOT NULL、FK、CHECKなどDB constraintで重要な不変条件を守っている。
- [ ] 同時更新、同時申込、同時購入などでrace conditionが起きない。
- [ ] 本番データを変更するAIエージェント操作には監査ログと承認がある。

## 7. AI・LLM特有の重大リスク

- [ ] Prompt Injectionによりシステム指示、開発者指示、ツール権限が上書きされない。
- [ ] RAGや外部ページの本文を「指示」としてではなく「データ」として扱っている。
- [ ] LLM出力をそのままSQL、shell、コード、URL、権限判断、決済判断に使っていない。
- [ ] AIが選んだtool callの引数をサーバー側でschema validationしている。
- [ ] AIが危険なtool、private data、他ユーザーデータへアクセスする前に権限確認している。
- [ ] AIのhallucinationを前提に、外部事実・金額・権限・状態変更は検証している。
- [ ] AIのJSON崩れ、空レスポンス、途中切れ、形式逸脱に耐えられる。
- [ ] AIの自己修正、retry、agent loopが無限に回らない。
- [ ] 高リスク出力には人間レビュー、二段階確認、または安全なfallbackがある。
- [ ] AIエージェントがユーザーに誤って法的・医療・金融・セキュリティ上の断定をしない設計になっている。

# P1 / High：本番前に原則対応する

## 8. 仕様適合・回帰防止

- [ ] 実装が元の要件、受け入れ条件、Issue、PR説明と一致している。
- [ ] AIが要件の曖昧な部分を勝手に確定していない。
- [ ] 既存仕様を削除・変更していない。変更した場合は理由が明記されている。
- [ ] 変更対象外の画面、API、DB、テスト、設定が不要に変わっていない。
- [ ] 既存機能の主要フローが壊れていない。
- [ ] 既存テストが通っている。
- [ ] 変更に対する回帰テストが追加または更新されている。
- [ ] AIが生成した仮実装、TODO、mock、placeholderが本番コードに残っていない。
- [ ] AIが作った名前、コメント、説明が実装内容と一致している。
- [ ] 仕様上の例外ケースがレビューされている。

## 9. 入力値・バリデーション

- [ ] フロントだけでなくサーバー側でもrequest body、query、paramsを検証している。
- [ ] 必須項目、型、長さ、範囲、形式、enumを検証している。
- [ ] 空文字、null、undefined、0件、未登録、初回利用時に壊れない。
- [ ] 長文、絵文字、改行、HTML、Markdown、特殊文字に対応している。
- [ ] ファイル名、URL、メールアドレス、電話番号などの形式検証が適切である。
- [ ] ユーザー入力由来の値を内部ID、role、price、plan、permissionとして信頼していない。
- [ ] AI出力由来の値をDB保存前・画面表示前・tool実行前に検証している。
- [ ] 不正入力時のエラーメッセージが過剰な内部情報を出していない。
- [ ] validation schemaがAPI、フォーム、型定義と乖離していない。
- [ ] 仕様上許可すべき値と拒否すべき値の境界が明確である。

## 10. API・通信

- [ ] HTTP methodが意味に合っている。
- [ ] 200、201、400、401、403、404、409、422、500などのstatus codeが適切である。
- [ ] APIレスポンス形式がフロントの期待と一致している。
- [ ] エラーレスポンスに扱いやすいcode、message、必要なmetadataがある。
- [ ] APIのbreaking changeが既存クライアントを壊さない。
- [ ] pagination、filtering、sortingが大量データ前提で設計されている。
- [ ] rate limit、quota、abuse対策が必要なAPIに入っている。
- [ ] 外部API呼び出しにtimeoutが設定されている。
- [ ] 外部API障害時にアプリ全体が停止しない。
- [ ] webhookは署名検証、重複処理、順不同到着に対応している。

## 11. 状態管理・整合性

- [ ] DB、キャッシュ、画面表示、外部サービスの状態が整合している。
- [ ] 楽観的更新後の失敗時にUIが正しい状態へ戻る。
- [ ] 作成・更新・削除後に関連キャッシュがinvalidateされる。
- [ ] 状態遷移が `pending`、`success`、`failed`、`cancelled` などで明確に管理されている。
- [ ] エラー時に中途半端な状態が残らない。
- [ ] ブラウザバック、リロード、再ログインで状態が破綻しない。
- [ ] localStorage、sessionStorage、cookieの利用目的と削除タイミングが明確である。
- [ ] 複数タブ操作で矛盾が起きにくい。
- [ ] queueやjobの再実行で同じ副作用が複数回起きない。
- [ ] AIエージェントの途中停止・中断・再開で状態が破綻しない。

## 12. 日付・時間

- [ ] UTC、JST、ローカルタイムの扱いが混在していない。
- [ ] DB保存時刻とユーザー表示時刻の変換が正しい。
- [ ] 期限、開始日、終了日、公開期間、予約時刻の境界が正しい。
- [ ] `start <= now < end` などの期間条件が明確である。
- [ ] 月末、年末、うるう年、月またぎで壊れない。
- [ ] 国際向けの場合、夏時間に対応している。
- [ ] cron、batch、schedulerがサーバーtimezone依存でズレない。
- [ ] 日付のoff-by-oneがない。
- [ ] 期限切れ表示や通知タイミングが仕様と一致している。
- [ ] AIが自然言語の日付を扱う場合、基準日・timezone・曖昧性の処理が明確である。

## 13. 決済・課金・有料機能

- [ ] 有料機能が無料ユーザーに開放されていない。
- [ ] plan、role、quotaの判定をフロントだけに依存していない。
- [ ] 決済ボタン連打、retry、webhook重複で二重課金されない。
- [ ] webhook処理が冪等である。
- [ ] 支払い成功、失敗、キャンセル、返金、解約、プラン変更の状態が正しく反映される。
- [ ] trial開始・終了の判定が正しい。
- [ ] test keyとlive keyが混ざっていない。
- [ ] 決済情報や請求情報を不要に保存していない。
- [ ] 課金状態のキャッシュが古いまま権限を与えない。
- [ ] AIエージェントが課金・返金・権限付与を自律実行する場合、人間承認または強い制限がある。

## 14. パフォーマンス・コスト高リスク

- [ ] N+1 queryが発生していない。
- [ ] 一覧表示や検索で大量データを一括取得していない。
- [ ] DB検索、join、sort、filterに必要なindexがある。
- [ ] 同じAPI、DB、LLM、外部サービスを無駄に複数回呼んでいない。
- [ ] payloadが不要に巨大化していない。
- [ ] 画像、動画、ファイルが適切に圧縮・遅延読み込みされている。
- [ ] LLMに不要な履歴、全文、重複contextを送っていない。
- [ ] 高額モデルを低価値処理に使っていない。
- [ ] retryに上限、backoff、circuit breakerがある。
- [ ] ユーザー単位・組織単位・IP単位などの利用上限がある。

## 15. テスト・CI

- [ ] 重要なビジネスロジックにunit testがある。
- [ ] 認証・認可・他人データアクセス不可のテストがある。
- [ ] API、DB、外部サービス連携のintegration testがある。
- [ ] 主要ユーザーフローのE2E testがある。
- [ ] 0件、1件、最大件数、境界値、異常入力のテストがある。
- [ ] エラー時、通信失敗時、外部API失敗時のテストがある。
- [ ] AI出力のJSON崩れ、空レスポンス、形式逸脱のテストがある。
- [ ] 外部APIやLLMをmockして安定したテストができる。
- [ ] lint、typecheck、testがCIで自動実行される。
- [ ] flaky testが放置されていない。

# P2 / Medium：安定運用・品質のために対応する

## 16. AIエージェント品質・評価

- [ ] AIエージェントの成功条件と失敗条件が定義されている。
- [ ] AIエージェントの出力品質を主観だけでなく評価基準で確認している。
- [ ] agent eval、golden dataset、regression prompt testなどがある。
- [ ] モデル変更時に品質・形式・コストが劣化しないか確認できる。
- [ ] promptの変更履歴が追跡できる。
- [ ] tool definition、schema、descriptionが曖昧でない。
- [ ] AIエージェントのmemoryに保存する情報と保存しない情報が定義されている。
- [ ] RAGの検索対象、権限、更新タイミング、削除反映が管理されている。
- [ ] AIが根拠を必要とする回答では参照元や根拠を扱える。
- [ ] 低信頼回答、不確実回答、危険回答ではfallbackや人間確認がある。

## 17. フロントエンド・UI/UX

- [ ] loading状態がある。
- [ ] error状態がある。
- [ ] empty状態がある。
- [ ] 保存・送信・削除などの成功/失敗フィードバックがある。
- [ ] 処理中にボタン連打できない。
- [ ] フォームの必須項目、形式、文字数、範囲が分かりやすい。
- [ ] PC、tablet、mobileで表示が崩れない。
- [ ] キーボード操作、focus、label、aria、contrastが最低限満たされている。
- [ ] SSR/CSR差分によるhydration errorがない。
- [ ] ユーザーがAIの処理中・完了・失敗・要確認を理解できる。

## 18. 保守性・設計

- [ ] 1つの関数が大きすぎない。
- [ ] 1つのコンポーネントが責務を持ちすぎていない。
- [ ] 1つのAPI routeに認証、DB、外部API、整形、AI処理が詰まりすぎていない。
- [ ] UI、ビジネスロジック、DB処理、外部API処理、AI処理が適切に分離されている。
- [ ] 同じ処理が複数箇所に重複していない。
- [ ] 命名が実態と一致している。
- [ ] magic number、固定ID、固定文字列、閾値が定数化されている。
- [ ] 設定値、モデル名、制限値、料金、API URLが一元管理されている。
- [ ] 循環依存や複雑な依存関係が増えていない。
- [ ] 過剰抽象化で読みにくくなっていない。

## 19. データベース・ストレージ

- [ ] schema、model、型定義、APIレスポンスが整合している。
- [ ] seed data、mock data、test dataが本番に混ざっていない。
- [ ] 重要データのbackupとrestore手順がある。
- [ ] データ保持期間、削除ポリシー、アーカイブ方針がある。
- [ ] 個人情報を必要以上に保存していない。
- [ ] 添付ファイル、生成物、ログ、キャッシュが無制限に増えない。
- [ ] storageの公開/非公開設定が正しい。
- [ ] 署名付きURLの有効期限が適切である。
- [ ] vector DBや検索indexに削除済みデータが残り続けない。
- [ ] DB migration後のbackfillやdata migrationが安全に設計されている。

## 20. 外部サービス連携

- [ ] 外部APIのscopeが最小権限である。
- [ ] OAuth scopeが過剰でない。
- [ ] 外部APIのrate limitに対応している。
- [ ] 外部APIの障害・遅延・部分失敗を想定している。
- [ ] 外部APIのレスポンスschema変更に脆すぎない。
- [ ] Webhookの順不同、遅延、重複、欠落に対応している。
- [ ] sandbox環境とproduction環境が明確に分かれている。
- [ ] API clientの認証情報更新・失効に対応している。
- [ ] 外部サービスから受け取るデータを信頼せず検証している。
- [ ] AIエージェントが外部サービス操作を行う場合、操作ログと承認導線がある。

## 21. 監視・ログ・運用

- [ ] 重要なエラーがログに残る。
- [ ] レイテンシ、エラー率、成功率、失敗理由を確認できる。
- [ ] AI API使用量、token量、コストを確認できる。
- [ ] 外部API使用量、rate limit到達、失敗率を確認できる。
- [ ] 重要操作のaudit logが残る。
- [ ] 本番でdebug logが過剰に出ていない。
- [ ] 個人情報やsecretがログに出ていないことを確認できる。
- [ ] health checkがある。
- [ ] 異常時のalertが設定されている。
- [ ] 障害時にAIエージェントを停止・制限できる。

# P3 / Low：保守性・開発体験を上げる

## 22. コード整理

- [ ] Dead code、未使用関数、未使用API、未使用componentが残っていない。
- [ ] 未使用import、未使用dependencyが残っていない。
- [ ] 古い分岐、仮実装、実験コードが残っていない。
- [ ] TODO、FIXME、HACKがIssueやタスクに紐づいている。
- [ ] コメントが実装と矛盾していない。
- [ ] README、セットアップ手順、環境変数一覧が更新されている。
- [ ] API仕様書、schema、型定義が更新されている。
- [ ] 変更理由、設計意図、代替案が必要な箇所に残っている。
- [ ] ディレクトリ構成が既存ルールと一貫している。
- [ ] AIエージェントが作成したファイル名・変数名が不自然でない。

## 23. 開発・CI/CD

- [ ] lockfileが正しく更新されている。
- [ ] Node、Python、Ruby、Goなどruntime versionが固定されている。
- [ ] local、CI、productionでbuild結果が再現できる。
- [ ] build artifactにsecretや内部情報が含まれていない。
- [ ] deploy前にlint、typecheck、test、security scanが走る。
- [ ] DB migrationとアプリdeployの順序がCI/CDで考慮されている。
- [ ] rollback手順がREADMEやrunbookにある。
- [ ] feature flagで段階公開できる。
- [ ] smoke testがdeploy後に実行される。
- [ ] preview deployの公開範囲が適切である。

## 24. 品質・チームレビュー

- [ ] AI生成コードであることを前提に、差分全体を人間がレビューしている。
- [ ] AIが修正したコードだけでなく、削除したコードもレビューしている。
- [ ] 関連する仕様書、Issue、テスト、ドキュメントも確認している。
- [ ] 変更範囲が大きすぎる場合、PRを分割できる。
- [ ] 命名、責務、抽象化、依存関係がチームの設計方針に合っている。
- [ ] レビューコメントに対してAIが場当たり的な修正を積み重ねていない。
- [ ] AIによる修正が新しいバグや別の仕様ズレを生んでいない。
- [ ] コードの読みやすさが担保されている。
- [ ] 将来の保守担当が理解できる構成になっている。
- [ ] 人間が最終責任を持つ前提でレビュー完了している。

# P4 / Ongoing：リリース後も継続確認する

## 25. 継続監視・改善

- [ ] リリース後のエラー率、クラッシュ率、問い合わせ、異常ログを確認している。
- [ ] AIエージェントの失敗例、誤回答、誤操作を収集している。
- [ ] AIエージェントの評価データセットを定期的に更新している。
- [ ] モデル変更、prompt変更、tool変更ごとに回帰評価している。
- [ ] 高コスト処理、token増加、外部API増加を定期的に確認している。
- [ ] 依存ライブラリの脆弱性を定期的に確認している。
- [ ] access log、audit logから不審な操作を確認している。
- [ ] ユーザー削除・データ削除がログ、cache、index、AI memoryへ反映されるか定期確認している。
- [ ] 障害対応runbook、復旧手順、連絡経路を定期的に見直している。
- [ ] AIエージェントの権限、利用可能tool、接続先を定期的に棚卸ししている。

# 最優先チェックだけを抜き出す場合

時間がないレビューでは、最低限以下だけは確認する。

- [ ] 認証・認可チェック漏れがない。
- [ ] 他人・他組織・他テナントのデータにアクセスできない。
- [ ] secret、個人情報、prompt/response logが漏れていない。
- [ ] AIエージェントが本番DB・本番secret・危険toolに無制限アクセスできない。
- [ ] Prompt Injectionやtool injectionで危険操作ができない。
- [ ] LLM出力をそのまま実行・保存・権限判断に使っていない。
- [ ] 既存機能が壊れていない。
- [ ] 要件と違うものを「完了」にしていない。
- [ ] 二重実行、二重課金、二重通知が起きない。
- [ ] migration、rollback、transactionが安全である。
- [ ] 本番/開発環境が混在していない。
- [ ] N+1、無駄なAPI呼び出し、AI APIコスト暴走がない。
- [ ] 重要ロジック、権限、エラー処理、AI出力検証にテストがある。
- [ ] エラー時にアプリやデータが中途半端な状態にならない。
- [ ] 人間が差分全体をレビューし、AI生成コードを鵜呑みにしていない。
