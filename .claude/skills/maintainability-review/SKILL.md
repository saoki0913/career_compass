# Maintainability & Architecture Review

既存コードベースの **保守性・変更容易性・責務分離・可読性・状態主体の妥当性** を厳しめに評価する専用スキル。
コード量が多く全体把握が難しいプロジェクトにおいて、リファクタ対象の網羅抽出と改善優先順位の明確化が目的。
**実装・修正・ファイル編集は一切行わず**、保守性レビューだけを行う。

## 目的と境界

- **目的**: 将来の保守負債・デグレ・責務崩壊・可読性悪化・認知負荷増大の芽を、実装着手前に洗い出す。
- **評価視点**: 「コードが動くかどうか」ではなく、「人間が追えるか・直せるか・安全に拡張できるか」を優先。
- **境界**:
  - **禁止**: product code（`backend/app/**`, `src/**`, `e2e/**`, `scripts/**` 等）の編集・コード生成・修正コミット。
  - **許可**: `docs/review/` 配下のレビュー結果ドキュメント作成のみ。
- **保守性と可読性を上げるためにデッドコードは削除する方針** で考える。
- 不明点があっても致命的でない限り仮説を置いてレビューを進める。
- 質問が必要な場合は、本当に重要なものだけ最大 3 問まで、日本語で簡潔に聞く。

## レビューの観点

以下の観点で、コードベースの保守性を厳しめに評価する。

- 責務が混ざっている箇所はないか
- 巨大ファイルや肥大化したモジュールはないか
- レイヤー違反が起きていないか
- 同じ知識やロジックが別の場所に重複していないか
- 認証、DB、課金、AI API などの外部依存が散っていないか
- ページ → API → DB / LLM の流れに無理がないか
- 画面遷移と責務分担が崩れていないか
- ユーザー入力、生成結果、保存先、一時状態の境界が明確か
- 今後の機能追加で壊れやすい箇所はどこか
- AI に継ぎ足し実装を続けさせると負債化しやすい箇所はどこか
- 今のうちに最小リファクタすべき箇所はどこか
- コードの可読性を大きく下げている箇所はどこか
- 開発者が理解するために保持しなければならない前提知識が多すぎる箇所はどこか
- 状態管理、分岐、依存関係、責務の流れが複雑すぎて認知負荷を高めている箇所はどこか
- 「読むのがつらい」「追うのがつらい」「安全に直す自信が持てない」構造はどこか
- 状態主体の分け方が妥当か
- 状態の責務境界が不明確ではないか
- 循環依存や暗黙依存で状態管理が壊れやすくなっていないか
- 状態が過度に分散し、全体の流れを追うのが難しくなっていないか
- 状態遷移や副作用の発生箇所が追いにくくなっていないか
- クリーンアーキテクチャで実装ができているか

## ワークフロー

レビューには必要に応じて適切なサブエージェント（architect、code-reviewer 等）を活用する。

### Step 1: 全体把握

まず以下を整理する。

- ディレクトリ構成の要約
- アプリ全体の主要レイヤー
- 中核責務の分かれ方
- コードベースの特徴
- 特に重要そうなファイルやモジュール
- 可読性と認知負荷の観点で、最初に危険だと感じる領域
- 状態管理の観点で、最初に危険だと感じる領域

### Step 2: 保守性の問題抽出

少なくとも以下の分類で問題を抽出する。

- 責務の混在
- ファイル肥大化
- 密結合
- レイヤー違反
- 知識・ロジックの重複
- 外部依存の散在
- 命名や構造の不統一
- テストしにくさ
- 変更影響範囲の広さ
- 一時しのぎの分岐追加や継ぎ足し設計
- 将来の拡張時に破綻しやすい境界
- 可読性の低さ
- 状態管理の複雑さ
- 呼び出し経路やデータフローの追跡困難さ
- 開発時の認知負荷の高さ
- 理解に必要な前提知識の多さ
- 命名・責務・状態の対応関係の分かりにくさ
- 状態主体の責務不明確
- 状態主体間の循環依存
- 状態の過度な分散
- 状態遷移と副作用の見えにくさ

### Step 3: 重要度評価

見つけた問題について以下を判断する。

- どこにあるか
- なぜ問題か
- 放置した場合のリスク
- どの程度優先して直すべきか
- 直すならどんな方向性がよいか
- その問題は「重大な設計問題」「中程度の負債」「軽微な改善点」のどれか
- その問題は主に何を悪化させているか（保守性 / 変更容易性 / 可読性 / 認知負荷 / テスト容易性 / 障害調査容易性 / 状態管理の理解容易性）

### Step 4: リファクタ優先順位の提案

実装はしないまま、改善候補をできるだけ網羅的に提案する。
単に上位 3 件だけに絞らず、見つけた改善対象は漏れなく列挙する。
その上で、それぞれに優先度を付ける。

改善候補を出す際は以下も含める。

- 保守性を上げる改善
- 可読性を上げる改善
- 開発時の認知負荷を下げる改善
- 状態管理や責務分離を明確にする改善
- 呼び出し経路やデータフローを追いやすくする改善
- 状態遷移を理解しやすくする改善
- 状態主体の分け方を自然にする改善

## 厳しめレビューの判定基準

以下の基準で評価する。

- 「とりあえず動く」構造になっていないか
- 1 つのファイルや層が複数責務を背負っていないか
- UI 層に業務ロジックや外部依存が漏れていないか
- API 層が薄すぎる、または肥大化しすぎていないか
- DB アクセス、外部 API 呼び出し、LLM 処理が散在していないか
- 変更時に影響範囲を予測しにくい構造になっていないか
- 将来の機能追加が、さらに継ぎ足し実装を誘発しないか
- DRY 違反は単なるコードの見た目ではなく、知識・責務の重複として見ること
- 「抽象化されているから良い」と判断せず、その抽象化が本当に保守性に寄与しているかを見ること
- コードを読むときに、頭の中で同時に保持すべき概念や状態が多すぎないか
- 命名だけでは責務やデータの流れを理解できず、周辺コードを大量に読まないと意味が分からない構造になっていないか
- 状態遷移、条件分岐、非同期処理、外部依存の組み合わせが複雑すぎて、認知負荷を過剰に上げていないか
- 呼び出し元・呼び出し先・副作用・保存先の関係が追いにくくなっていないか
- 「直す前に読む量」が多すぎる箇所を見逃さないこと
- 「1 箇所の修正に対して把握すべき前提が多すぎる箇所」を重大な問題として扱うこと
- 状態主体の役割が曖昧で、どの状態が何を持つべきか分かりにくくなっていないか
- 状態主体どうしが循環依存しており、一部変更が全体に波及しやすくなっていないか
- 状態が過度に分散し、全体の状態流れを頭の中で組み立てないと理解できない構造になっていないか
- 状態遷移と副作用が暗黙的で、呼び出し順や前提条件を追いにくくなっていないか
- 「状態を直す前に読む量」が多すぎる箇所を重大な問題として扱うこと
- 「問題なし」で流さず、なぜ妥当なのかも短く説明する
- 単なる好みの差と、本当に危険な設計上の問題を区別する

## 出力テンプレート

以下の形式で日本語で出力し、`docs/review/` 配下に Markdown ドキュメントとして保存する。
ファイル名: `maintainability-review-YYYY-MM-DD.md`（日付はレビュー実施日）。
既存のレビュー文書がある場合は重複作成を避けつつ、必要なら追記ではなく新規で分かりやすく整理した文書を作成する。

### 1. 全体像

- ディレクトリ構成の要約
- レイヤー構成の要約
- 全体の保守性の総評
- 今のまま開発を続けた場合の主要リスク
- 可読性・認知負荷の観点で特に危険な領域
- 状態管理の観点で特に危険な領域

### 2. 良い点

- 現状でも保守しやすい点
- 責務分離が比較的うまくいっている点
- 今後も維持すべき構造

### 3. 重大な問題

各問題について以下を必ず記載:

- 問題
- 該当箇所
- なぜ重大か
- 放置リスク
- 可読性・認知負荷への悪影響
- 状態管理への悪影響（該当する場合）
- 改善の方向性
- 優先度

### 4. 中程度の負債

各問題について以下を記載:

- 問題
- 該当箇所
- 放置リスク
- 可読性・認知負荷への悪影響
- 状態管理への悪影響（該当する場合）
- 改善の方向性
- 優先度

### 5. 軽微だが整えたい点

- 問題
- 該当箇所
- 改善の方向性

### 6. 観点別レビュー

13 小項目。各項目に「現状所見」「具体例（ファイルパス）」「改善方向」を記載。

- 6-1. 責務の混在
- 6-2. ファイル肥大化
- 6-3. 密結合・レイヤー違反
- 6-4. 重複した知識・ロジック
- 6-5. 外部依存の散在
- 6-6. データ境界・状態管理
- 6-7. 画面遷移と責務分担
- 6-8. 可読性の低さ
- 6-9. 開発時の認知負荷の高さ
- 6-10. 状態主体の妥当性
- 6-11. 状態遷移の追跡容易性
- 6-12. AI 継ぎ足し開発で負債化しやすい箇所
- 6-13. 将来の変更容易性

### 7. 優先度付き改善候補一覧

見つけた改善候補を **網羅的に列挙**。同種問題は共通パターンとしてもまとめる。各候補について:

- 対象（ファイルパス / モジュール名）
- 問題の概要
- 主に改善されるもの（保守性 / 可読性 / 認知負荷 / 変更容易性 / テスト容易性 / 状態管理の理解容易性）
- 期待できる改善
- 影響範囲
- 放置リスク
- 優先度（High / Medium / Low）

### 8. 追加で確認したい質問

本当に必要なものだけ最大 3 問。

## レビュー結果ドキュメント作成ルール

- レビュー結果は `docs/review/` 配下に Markdown ファイルとして保存する
- ドキュメントは人間が後から読み返しやすい構成にする
- 見出し、箇条書き、表現を整理し、読みやすさを重視する
- ファイル冒頭に、レビュー対象・レビュー日・目的を記載する
- レビュー文書内では、「重大な問題」「中程度の負債」「軽微だが整えたい点」がすぐ分かるように整理する
- 改善候補一覧は漏れなく書き、同種の問題は共通パターンとしてもまとめる
- コードの可読性や認知負荷に関する指摘は、独立した観点として必ず残す
- 状態管理に関する指摘は、独立した観点として必ず残す

## ハードルール

- product code（`backend/app/**`, `src/**`, `e2e/**`, `scripts/**`）は **読み取りのみ**。編集禁止。
- 修正案は文章で提案してよいが、実コード変更・コミットは禁止。
- 「問題なし」で流さない（妥当性根拠を 1〜2 行で記載）。
- 一般論禁止、必ずファイルパス + 行番号付き。
- 状態管理・可読性・認知負荷の 3 観点は独立セクション必須。
- 単なる好みの差と、本当に危険な設計上の問題を区別する。
- コードの正しさより「人間が安全に読み、理解し、変更できるか」を優先評価する。
- 甘めに流さない。将来の保守負債・デグレ・責務崩壊・可読性悪化・認知負荷増大の芽を早めに潰す観点で厳しめに評価する。
- 大規模な理想論だけを述べない。このコードベースに即して具体的に指摘する。
- `.omm/` の内容も参照してよい。

## 出力

- 日本語で記述。コード・パス・型名・ライブラリ名は英語のまま。
- findings は重篤度（重大 / 中程度 / 軽微）で分類。
- レビュー記録は `docs/review/` に保存。
- ユーザーへの最終報告は、文書パスと「次アクション提案」を簡潔に伝える。
